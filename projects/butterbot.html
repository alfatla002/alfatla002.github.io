Project

ButterBot — Discord Economy Bot

Overview

ButterBot is a modular Discord bot designed to simulate a persistent economy shared between users within a server. The project began as a simple command-based bot but evolved into a structured backend system capable of handling financial interactions such as loans, inventory management, shop purchases, and user protection mechanics.

The goal of the project was to design a system that could handle complex interactions between users while maintaining data consistency, scalability, and maintainable code structure. It also served as an exploration of backend architecture decisions, particularly the transition from document-based storage to relational database design.

System Design and Development Process

The initial version of the bot used MongoDB for data storage. While this allowed rapid development, limitations became apparent when implementing relationships between users, particularly for loan agreements and shared economic interactions. Queries became increasingly complex and difficult to maintain.

To address this, the system was redesigned around a relational SQL schema. Tables were introduced for users, balances, loan agreements, bodyguards, and theft statistics. This allowed relationships between users to be enforced at the database level and simplified transactional updates.

The application structure was refactored into modular components, separating economy logic from command handling and UI behavior. Discord slash commands and interactive UI elements were introduced to reduce command complexity and improve usability.

Shop items were implemented as independent logic modules, enabling future expansion without modifying core bot logic and reducing the risk of breaking existing functionality.

Technologies Used

  • Node.js
  • Discord.js
  • SQL database
  • Slash commands and Discord UI components
  • Modular architecture design

Engineering Challenges & Solutions

A major challenge was maintaining consistency between user balances during concurrent operations such as loans or theft attempts. Moving to SQL enabled transactions and structured updates that prevented inconsistent states.

Another challenge involved balancing gameplay mechanics so that the economy remained stable over time. Cooldowns and limits were introduced to prevent abuse while maintaining engagement.

Refactoring an existing project without breaking functionality required careful testing and incremental changes, reinforcing the importance of modular design.

Outcome

The final system demonstrates backend architecture principles, relational database design, and modular application development. The project highlights the transition from rapid prototyping toward maintainable, scalable software engineering practices.
