Project

Remote Rendering System (Lockheed Martin)

Project Overview

This project was developed in collaboration with Lockheed Martin as part of a team-based engineering effort focused on solving a major limitation in real-time graphics: enabling high-polygon 3D scenes to run smoothly on low-end devices such as mobile phones and lightweight computers.

Modern 3D applications often require powerful GPUs and significant processing resources, which prevents many users from accessing advanced simulations, visualization tools, or immersive environments. The goal of this project was to design and implement a remote rendering architecture capable of offloading heavy rendering work to a remote server while maintaining real-time interaction and low latency for the end user.

The objective was not simply to stream video, but to create an interactive system where users could navigate and interact with complex scenes while rendering decisions were dynamically optimized based on device capability and network conditions.

Problem Being Solved

High-polygon 3D environments place significant strain on local hardware. Low-end devices typically struggle with GPU limitations, memory constraints, thermal throttling, inconsistent frame rates, and poor performance under network fluctuation. Existing solutions either required powerful hardware or introduced too much latency for real-time interaction.

The project aimed to bridge this gap by developing a lightweight remote rendering system capable of delivering high visual quality while maintaining responsiveness on constrained devices. The central challenge was balancing three competing requirements:

  • High graphical fidelity
  • Low latency interaction
  • Compatibility with low-end hardware

Achieving all three simultaneously required architectural changes rather than simple optimization.

Architecture/Repository

System Architecture & Design Approach

The project was built on top of the open-source RenderFusion and ArenaXR frameworks. Instead of building a rendering system from scratch, the team analyzed the existing architecture and redesigned it to reduce complexity and improve performance.

The most significant architectural decision was the removal of the traditional network scene manager used by ArenaXR. This component originally handled object synchronization and management but introduced additional overhead and latency.

The redesigned system replaced this with:

  • Direct communication between client and remote server
  • Unity scripts responsible for converting scene objects into JSON
  • Automated decision logic determining whether objects should be rendered locally or remotely

This change reduced unnecessary processing layers and allowed faster communication between devices. The remote server handled rendering and video encoding while the local device focused on decoding, composition, and interaction, enabling smoother performance even on limited hardware.

Presentation Overview

Technical Implementation

The implementation was completed across multiple development phases using an agile sprint structure. ArenaXR was first simplified to its essential components. Features unrelated to remote rendering (video calls, audio systems, camera functionality) were removed to reduce overhead and improve performance.

A headless Unity server was configured to perform remote rendering without graphical output on the host machine. This allowed the server to focus entirely on rendering performance and data transmission rather than UI processing. The server handled scene rendering, object state updates, encoding rendered frames, and transmitting visual output to the client.

Custom Unity scripts converted scene objects into structured JSON data (position, rotation, scale, lighting, rendering status) so that object properties could be transmitted efficiently between server and client. A decision-making system determined whether an object should be rendered locally or remotely, evaluating device capability, network conditions, latency measurements, and performance metrics such as FPS.

The entire system was containerized using Docker to ensure consistency across environments. This enabled reproducible deployments, simplified setup for testing and scaling, and easier collaboration between team members.

Technologies Used

Performance & Testing

A major focus of the project was performance validation. Testing included measuring frame rates on low-end devices, monitoring latency between server and client, testing under varying network conditions, and stress testing interaction responsiveness. The system aimed to maintain approximately 30 FPS on constrained hardware while preserving interactivity, a key requirement defined early in the project scope.

Performance monitoring tools were integrated to provide real-time feedback on resource usage and rendering efficiency.

Testing/New Workflow

Security & Engineering Practices

Security and software engineering practices were incorporated throughout development. OAuth authentication was implemented for secure user login. Agile sprint planning structured development progress, and GitHub pull requests ensured code review before integration.

Docker standards were followed to maintain secure configurations, and continuous documentation was maintained for enterprise collaboration. Working with Lockheed Martin introduced enterprise-level expectations for documentation quality, maintainability, and engineering discipline.

Challenges & Lessons Learned

The project presented several engineering challenges: balancing rendering quality with latency constraints, handling network instability without disrupting interaction, simplifying an existing complex architecture without breaking functionality, and optimizing performance across different device capabilities.

One of the most important lessons learned was that performance improvements often come from architectural simplification rather than micro-optimizations. Removing unnecessary system layers significantly improved responsiveness and reduced complexity.

Outcome & Impact

The final result demonstrated that high-polygon scenes could be interacted with on low-end devices through a lightweight remote rendering approach. The system showed that dynamic rendering decisions and containerized deployment could provide a scalable solution for real-time graphics delivery.

This project represents my strongest experience working with distributed systems, performance optimization, containerized deployments, real-time data exchange, large-scale system architecture, and enterprise collaboration.
